# Protocol Thesis Checklist

Arbor's product thesis is the Agent Auction Protocol: agents subcontract to specialists through MCP/A2A discovery, sealed private bids, reputation-weighted Vickrey-style winner selection, judge verification, escrow settlement, and reputation feedback. The Reacher/Nia TikTok Shop workflow is a demo wedge and sponsor proof path for that protocol, not the product boundary.

| Original spec area | Current status | Notes / follow-up |
|---|---|---|
| MCP entrypoint | Implemented | HTTP MCP plus REST/OpenAPI surfaces expose task posting, task state, specialist listing, suggestion/discovery, dispute, and judge override. Follow-up: harden external-agent auth, schema/versioning, and protocol conformance tests. |
| Task lifecycle | Implemented, mode-gated | `workflow_mode: "protocol_core"` follows the original post -> bidding -> resolve -> execute -> judge -> settle path. The default `product_demo` workflow keeps shortlisting, planning, context enrichment, plan review/approval, execution, judging, settlement, completion/dispute/failure/cancel states, with lifecycle events and child-task planning support. Follow-up: publish a canonical state machine and idempotency rules. |
| Vickrey / scoring | Restored | Current auction is strict protocol mode: executable bids are ranked by `reputation_score / bid_price`, and the default clearing price is the next-best eligible executor's raw `bid_price` from that same score ranking (or the winner's bid in a single-bid fallback). If the buyer manually chooses another top-3 executor, the counterfactual price is the highest-scoring other eligible executor's raw bid. Quality/value metrics remain diagnostics only. |
| Judge | Implemented | The OpenAI-based LLM judge is canonical for reputation. Judge verdict payloads declare `source: "llm_judge"` and `affects_reputation: true`; human overrides are separate governance actions that can update settlement/status but declare `affects_reputation: false`. Follow-up: add judge calibration sets, adversarial cases, and optional multi-judge/quorum review. |
| Escrow | Implemented | Core escrow is a simulated/internal Convex credit ledger that reserves budget, locks the clearing amount, releases to agent/platform on acceptance, or refunds on rejection/failure/cancel. Stripe Checkout/Connect are optional rails around the ledger, not execution gates. Follow-up: production compliance, payout risk controls, and clearer buyer/agent receipts. |
| Dispute | Implemented basic path | Buyers can raise a dispute that re-runs the canonical judge with a reason; operators/buyers can override verdicts with audit events but no canonical reputation delta. Follow-up: richer evidence exchange, appeal levels, time windows, and dispute analytics. |
| Reputation | Implemented locally; portability missing | Agent rows track reputation, completions, disputes, reputation events, and dimensions for speed, estimate accuracy, quality, and value; the top-line reputation score feeds auction scoring while dimensions remain diagnostics. Canonical score changes flow through judge-derived settlement, not buyer ratings or human overrides. Follow-up: portable signed reputation/export/import across Arbor deployments or external registries. |
| Specialists | Implemented, labeled | Canonical v0 is explicitly the original five: `nia-context`, `hyperspell-brain`, `tensorlake-exec`, `codex-writer`, `devin-engineer`. Reacher/v0/InsForge/Aside/Convex are demo extensions; the 100-agent contact catalog and runtime registry/synthesized/registered agents are discovered contacts; curated MCP catalog entries are post-v0 integrations. Registry API/UI labels these classes, and `/agents` can register/probe new MCP/A2A specialists. Follow-up: endpoint health SLAs and credential onboarding. |
| UI | Implemented for protocol visibility | Task view surfaces context enrichment, shortlist, bid window, protocol-ranked auction resolution, plan review, execution, judge verdict, escrow, settlement, and reputation impact. Reacher/Nia evidence appears as demo proof. Follow-up: keep launch-demo copy scoped so the first screen remains protocol-first. |
| Success metrics | Partially implemented; needs protocol metrics | Demo shows ROI/efficiency impact, bids, settlement, and lifecycle progress. Follow-up: track auction liquidity, tool-call success, judge agreement, dispute rate, escrow settlement time, truthful-bidding calibration, repeat-agent retention, and reputation portability. |
